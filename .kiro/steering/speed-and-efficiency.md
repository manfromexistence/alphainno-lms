# Execution Speed & Efficiency Rules

## CRITICAL: No Summary Documents
- **NEVER** create summary markdown files (e.g., `SUMMARY.md`, `IMPLEMENTATION_STATUS.md`, `COMPLETE.md`)
- **NEVER** write lengthy documentation about what was accomplished
- **NEVER** create status tracking files unless explicitly requested by the user
- These waste time and provide no value to the user

## Response Style
- Keep final responses **extremely brief** (1-3 sentences max)
- State only what was done, not how or why
- Example: "Fixed the duplicate method error in OnlineExamController."
- **NO** bullet point lists in final summaries
- **NO** verbose explanations of the work process

## Execution Priority
1. **Read the user's request carefully**
2. **Execute the task immediately** - no planning documents
3. **Make the changes** - code first, talk later
4. **Verify it works** - test if needed
5. **Respond briefly** - one sentence is enough

## What TO Do
- Focus on **speed** and **accuracy**
- Make direct code changes immediately
- Use tools efficiently (parallel operations when possible)
- Test only when necessary
- Give minimal confirmation when done

## What NOT To Do
- ❌ Create summary documents
- ❌ Write long explanations
- ❌ Document the process
- ❌ Create status files
- ❌ Provide detailed recaps
- ❌ List everything that was changed
- ❌ Explain reasoning unless asked

## Exception
Only create documentation files when the user **explicitly requests** them (e.g., "create a README", "document this feature").
